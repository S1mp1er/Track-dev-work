# track-dev-work

一个面向 Codex 的个人研发过程记录 Skill。

它不记录每轮对话，也不生成流水账，而是只沉淀少量高价值节点，用尽可能少的记录还原完整研发流程，并在结束时输出可复盘、可写简历的总结。

## 适合什么场景

- 实习或正式开发中，自己不会或者懒得整理需求，但又希望每天快速回顾自己做了什么
- 做完需求后，想把开发目标、关键过程、解决方法和结果沉淀下来
- 排查 Bug、做重构、做联调时，希望最后能直接提炼成简历亮点
- 不想维护一份冗长的工作日志，只想保留真正重要的节点

## 核心特点

- 只记录关键节点，不按对话轮次记录
- 一次需求一份 Markdown，历史只追加，不自动改写旧记录
- 支持多个并行需求，通过稳定记录 ID 区分
- 自动把 `work-records/` 写到 Git 项目的 `.git/info/exclude`
- 完成时自动生成完整流程总结
- 输出 1 到 2 条保守、可验证的简历描述
- 明确禁止虚构性能、规模和业务指标

## 记录策略

默认只保留下面几类高价值节点：

- 开发目标
- 关键判断
- 根因与方向
- 解决方案
- 关键结果
- 最终总结

普通需求通常只需要：

- 1 条启动记录
- 0 到 2 条重要过程节点
- 1 条最终总结

即使任务复杂，自动过程节点也最多只保留 3 条。目标是让你在回看时，能通过几个节点就还原整个工作流程，而不是重新阅读一整天的对话。

## 安装

### 方式一：直接克隆到本地技能目录

```bash
git clone https://github.com/<your-name>/track-dev-work.git ~/.codex/skills/track-dev-work
```

### 方式二：下载后手动放入技能目录

把整个目录放到：

```bash
~/.codex/skills/track-dev-work
```

## 使用方式

在 Codex 中明确发出开始记录的指令，例如：

```text
使用 $track-dev-work 开始记录这个需求
```

或：

```text
开始记录这个需求
```

后续只有在真正关键的节点，Skill 才应该追加内容。比如：

- 开发目标被明确
- 根因被确认
- 方案和取舍稳定下来
- 一个关键可验证结果已经产生
- 你明确说“记录一下”“阶段总结”“完成记录”

完成时可以这样说：

```text
完成记录
```

或：

```text
总结并完成
```

## 结束时会输出什么

完成后，Skill 会基于整份记录输出一份完整总结，包含：

- 开发/工作目标
- 工作流程复原
- 遇到的问题
- 中间分析与方向
- 解决方法
- 最终结果与验证
- 整体亮点
- 多维度简历价值评分
- 1 到 2 条可直接改写进简历的描述

其中“解决方法”会被重点展开，因为它通常最能体现你的分析能力和工程价值。

## 日志文件行为

- 默认在项目根目录创建 `work-records/`
- Git 项目下只修改 `.git/info/exclude`，不会改团队 `.gitignore`
- 文件名格式：

```text
<需求标题>_<YYYYMMDD-HHmmss>_<进行中|受阻|待验证|已完成>.md
```

- 状态变化只重命名文件并追加事件，不删除历史内容
- 不提供删除命令

## 约束与边界

- 不记录密码、令牌、密钥和个人敏感信息
- 不虚构性能、规模、收益或业务指标
- 缺失的量化证据会保留为 `TODO：补充可验证指标`
- 简历候选内容默认泛化内部系统名和保密信息

## 示例价值

这个 Skill 更像一个“研发复盘压缩器”，而不是聊天日志工具。

它的目标不是记得更多，而是用最少的关键节点，帮你在事后快速回答这些问题：

- 我这次需求的目标是什么
- 中途真正卡在哪里
- 我是怎么分析并找到方向的
- 最终用了什么方法解决
- 结果是否被验证
- 哪些内容值得写进简历
